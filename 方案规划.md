# 智能计算系统大作业：Qwen2 + 手写 Torch RAG 方案规划

## 1. 结论：方案是否可行

结论：当前方案总体可行，并且和任务书约束匹配。

推荐主线方案为：

1. 使用本地 `model/qwen_model/` 中的 **Qwen2-0.5B (bf16)** 模型（实测 `config.json` 显示 `model_type: qwen2`、`architectures: Qwen2ForCausalLM`、权重文件 942MB），作为回答生成模型。
2. 使用本地 `model/bge_model/` 中的 `BGE-small-zh-v1.5` 作为查询和知识片段编码模型。
3. 离线阶段把课程知识 JSON 切分为片段，并生成：
   - `model/data/knowledge.json`：原始或格式化知识数据。
   - `model/data/embeddings.npy`：每个知识片段的向量。
   - `model/data/metadata.json`：每个向量对应的文本、标题、来源、标签等元信息。
4. 推理阶段只运行 `main.py`，流程为：
   - 从 `sys.argv[1]` 读取问题。
   - 离线加载 BGE 模型，编码问题向量。
   - 用 `torch` 手写余弦相似度或点积相似度，和 `embeddings.npy` 检索 Top-K 知识。
   - 把检索知识注入 prompt。
   - 离线加载 Qwen2，生成最终答案。
   - `print()` 输出非空回答。

该方案符合任务书中允许使用开源模型、RAG、辅助数据文件的要求；也符合禁止外部 API、禁止运行时联网下载、压缩包中唯一 `main.py` 的要求。

## 2. 任务书关键约束

从 `智能计算系统大作业任务书.md` 可归纳出以下硬约束：

1. 提交内容必须是 `.zip`。
2. 压缩包中必须能找到唯一的 `main.py`。
3. 系统会执行：

```bash
python3 main.py "问题文本"
```

因此 `main.py` 必须：

- 从 `sys.argv[1]` 读取问题；
- 不使用 `input()`；
- 不把答案写入文件作为唯一输出；
- 使用 `print(...)` 把最终答案输出到标准输出；
- 输出不能为空；
- 程序应正常结束，不能报错退出；
- 不要向 stdout 打印大量调试信息。

4. 评测机环境为：

| 项目 | 版本 |

| Python | 3.12.3 |
| CUDA 驱动 | 支持 CUDA 13.0 |
| PyTorch CUDA | 12.8 |
| cuDNN | 9.10.2 |
| PyTorch | 2.8.0+cu128 |
| Transformers | 5.6.2 |

5. 运行时不要联网，不要在代码中自动下载模型或数据。
6. 尽量只使用评测环境已有库或标准库。
7. 上传 ZIP 文件大小限制为 5GB 以内。

## 4. 主线代码依赖边界：必须严格区分推理与离线构建

这是本方案最重要的工程约束。

### 4.1 `main.py` 推理阶段允许使用的库

`main.py` 是评测机直接运行的入口，因此必须严格限制依赖。

主线 `main.py` 只允许依赖：

- Python 标准库：
  - `sys`
  - `os`
  - `json`
  - `pathlib`
  - `traceback`，仅限异常降级时内部使用，不打印到 stdout
  - `typing`
- 评测环境明确存在或高概率随模型推理环境存在的库：
  - `torch`
  - `transformers`
  - `numpy`

`main.py` 不应依赖：

- `sentence_transformers`
- `FlagEmbedding`
- `faiss`
- `langchain`
- `accelerate` 的显式 API
- `modelscope`
- `scikit-learn`
- `pandas`
- 任意联网下载逻辑

注意：`transformers` 内部可能间接调用 `safetensors` 读取模型权重，这是合理的；但 `main.py` 中一般不需要显式导入 `safetensors`。

### 4.2 离线构建阶段可以使用更多库

生成 `.npy` 的脚本属于离线预处理阶段，不在评测机主线运行。

离线构建阶段可以根据本地开发便利使用更多库，例如：

- `tqdm`
- `sentence_transformers`
- `FlagEmbedding`
- `pandas`
- `jsonlines`
- `modelscope`

但是最终提交时必须保证：

1. `main.py` 不导入这些额外库。
2. 评测运行不需要执行离线构建脚本。
3. 已经生成好的 `embeddings.npy`、`metadata.json`、`knowledge.json` 随 ZIP 一起提交。
4. 如果提交保留离线脚本，也不能让 `main.py` 调用它们。

## 6. 主要风险与应对

### 6.1 Qwen2 架构兼容风险（降低）

`model/qwen_model/config.json` 中显示：

- `model_type`: `qwen2`
- `architectures`: `Qwen2ForCausalLM`
- `transformers_version`: `5.8.0`

任务书评测机为 `Transformers 5.6.2`，本地 `ourLLM_env` 也是 `5.6.2`。config 要求的 5.8.0 **略高于**评测机版本。

Qwen2 是 transformers 原生支持的稳定架构，5.6.2 大概率可以加载。但与 5.8.0 的差异需实际测试。

风险较原预估降低，因为：
1. Qwen2 是成熟架构，非实验性 Qwen3.5。
2. 权重使用标准 bf16 格式，无需量化后端。
3. 单文件 `model.safetensors`，无需 index / shard 加载逻辑。

应对：

1. 第一阶段先写最小 `main.py`，只加载 Qwen2 并回答一句。
2. 使用 `local_files_only=True` 禁止联网。
3. 若 transformers 5.6.2 报错，检查具体是哪条 API 不兼容，针对性降级配置（如移除 5.8.0 才支持的字段）。

### 6.2 量化兼容风险（已消除）

原计划担心 AWQ / compressed-tensors 需要额外后端支持。实测 `model/qwen_model/config.json` 中 **没有** `quantization_config` 字段，`dtype` 为 `bfloat16`，权重为普通 safetensors 格式。

结论：**该风险不存在，无需应对。** 直接使用 `AutoModelForCausalLM.from_pretrained` 加 `torch_dtype="auto"` 即可加载，无需量化后端。

### 6.3 ZIP 大小风险

任务书要求 ZIP 小于 5GB。

当前 Qwen 权重 942MB（bf16），BGE-small 很小（~200MB），RAG `.npy` 取决于片段数量。三者合计远小于 5GB，空间充裕。

建议：

1. 控制 RAG 片段数量，例如 200～2000 条。
2. BGE-small 向量维度为 512，float32 每条约 2KB，2000 条约 4MB，可接受。
3. `metadata.json` 不要塞入过长全文，知识片段每条建议 200～800 中文字。
4. 打包前执行：

```bash
du -sh .
zip -r model.zip main.py model rag
ls -lh model.zip
```

### 6.4 时间与空间风险

模型约 0.5B 参数（bf16），加载速度较快。但每次推理都重新启动进程并加载模型，时间仍应控制。

应对：

1. `main.py` 尽量简洁，避免运行时构建索引。
2. 预生成 `.npy`，运行时只 `np.load`。
3. BGE 模型小，加载快；Qwen2 0.5B 即使 CPU 加载也很快，GPU 更佳。
4. `max_new_tokens` 建议控制在 384～768，避免过慢。

## 8. JSON 数据格式占位设计

用户希望 JSON 具体格式暂时留空可补充。建议采用兼容性强、易切块的列表格式：

```json
[
  {
    "id": "",
    "title": "",
    "source": "",
    "tags": [],
    "question": "",
    "answer": "",
    "content": ""
  }
]
```

字段建议：

| 字段 | 用途 |
|---|---|
| `id` | 唯一编号，例如 `lab1_q1` |
| `title` | 知识主题，例如 `卷积层正确性验证` |
| `source` | 来源，例如 `实验一思考题` |
| `tags` | 标签，例如 `CNN`、`正确性验证` |
| `question` | 如果原始材料是问答，可存题目 |
| `answer` | 如果原始材料是问答，可存标准答案或整理答案 |
| `content` | 主要知识正文 |

离线构建时每条最终用于 embedding 的文本可拼成：

```text
标题：{title}
问题：{question}
内容：{answer 或 content}
标签：{tags}
```

## 9. 分阶段实施规划

### 阶段 1：最小 Qwen 启动版 `main.py`

目标：只验证 Qwen2 能否在评测环境等价环境中离线加载并输出。

实现要点：

1. `sys.argv[1]` 读取问题。
2. 使用 `Path(__file__).resolve().parent` 定位模型目录。
3. 加载：

```python
AutoTokenizer.from_pretrained(QWEN_DIR, local_files_only=True)
AutoModelForCausalLM.from_pretrained(QWEN_DIR, local_files_only=True, torch_dtype="auto")
```

4. 优先手动设备选择，降低对额外库的依赖：

```python
device = "cuda" if torch.cuda.is_available() else "cpu"
model.to(device)
```

5. 使用 `tokenizer.apply_chat_template()` 构造对话。如果模型 tokenizer 支持 chat template，优先使用。
6. `max_new_tokens` 初始设为 512。
7. stdout 只打印最终答案。
8. 异常时输出兜底回答，保证非空。

验证命令：

```bash
micromamba run -n ourLLM_env python main.py "简单介绍一下自己"
```

### 阶段 2：离线 BGE 编码与 `.npy` 生成

目录：`rag/build_embeddings.py`

该脚本可以使用更多库，但建议仍用 `torch + transformers + numpy` 实现，保证训练/构建流程也足够简单。

流程：

1. 读取 `model/data/knowledge.json`。
2. 将每条 JSON 转成待编码文本。
3. 加载 `model/bge_model/`：

```python
AutoTokenizer.from_pretrained(BGE_DIR, local_files_only=True)
AutoModel.from_pretrained(BGE_DIR, local_files_only=True)
```

4. 对文本 batch tokenize。
5. BGE-small 常见做法：取 `[CLS]` 向量并归一化：

```python
outputs = model(**inputs)
embeddings = outputs.last_hidden_state[:, 0]
embeddings = torch.nn.functional.normalize(embeddings, p=2, dim=1)
```

6. 保存：

```python
np.save(DATA_DIR / "embeddings.npy", embeddings.cpu().numpy().astype("float32"))
```

7. 生成 `metadata.json`，保持和 `.npy` 行号一一对应。

### 阶段 3：手写 Torch 相似度检索

目录：`rag/retriever.py` 或直接内联到 `main.py`。

为避免主线导入路径复杂，最终建议把必要检索函数内联到 `main.py`，`rag/retriever.py` 仅作为开发版。

核心算法：

1. 加载 `.npy`：

```python
emb = torch.from_numpy(np.load(EMBEDDINGS_PATH)).to(device)
```

2. 如果离线保存时已经归一化，则查询向量也归一化后直接点积：

```python
scores = torch.matmul(emb, query_emb)
values, indices = torch.topk(scores, k=min(top_k, emb.shape[0]))
```

3. 如果无法保证归一化，则使用余弦相似度：

```python
emb_norm = torch.nn.functional.normalize(emb, p=2, dim=1)
query_norm = torch.nn.functional.normalize(query_emb, p=2, dim=0)
scores = emb_norm @ query_norm
```

4. 根据 `metadata.json` 的索引取回知识片段。

### 阶段 4：完整 RAG `main.py`

主流程：

```text
启动 main.py
  ↓
读取 sys.argv[1]
  ↓
加载 Qwen tokenizer/model
  ↓
尝试加载 RAG：BGE tokenizer/model + embeddings.npy + metadata.json
  ↓
编码问题向量
  ↓
手写 torch 相似度 Top-K
  ↓
构造知识上下文
  ↓
构造最终 prompt
  ↓
Qwen 生成答案
  ↓
print(answer)
```

RAG 加载失败时：

```text
跳过 RAG，直接用 Qwen 根据问题回答
```

Qwen 加载或生成失败时：

```text
输出保底答案：抱歉，我暂时无法完整回答这个问题，但可以从模型实现正确性、性能分析和实验验证角度进行分析。
```


公开集问题多为课程思考题，建议回答风格：

2. 再分点说明方法。
3. 最后给工程实践建议或注意事项。

## 11. RAG 检索参数建议

初始参数：

| 参数 | 建议值 |
|---|---|
| `top_k` | 3 |
| 单条知识最大字符数 | 800 |
| 注入上下文最大字符数 | 2500～4000 |
| BGE 输入最大长度 | 512 tokens |
| Qwen 最大输入长度 | 可先限制为 4096～8192 tokens |
| `max_new_tokens` | 512 |
| `temperature` | 0.2～0.6 |
| `top_p` | 0.8～0.95 |

如果追求稳定性和课程问答准确性，建议低温：

```python
do_sample=False
```

或：

```python
do_sample=True
temperature=0.3
top_p=0.9
```

## 14. 最终判断
