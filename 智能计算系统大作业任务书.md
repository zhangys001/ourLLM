# 1. 作业背景
随着大语言模型在问答、推理、代码生成、知识检索和智能体系统中的广泛应用，如何在有限硬件资源下构建一个**效果好、速度快、资源占用低、可稳定运行**的自然语言系统，已经成为智能计算系统中的重要问题。

本次期末大作业采用打榜形式。各位同学需要提交自己的自然语言生成模型及对应的调用文件 `main.py`。评测系统会自动运行同学提交的模型，并从**回答质量、运行时间、空间占用、系统稳定性**等方面进行综合评价和排名。

# 2. 作业目标
本次大作业要求同学们围绕给定评测环境，设计并实现一个可自动运行的自然语言模型问答系统。系统可以基于开源大模型，也可以在此基础上进行微调、后训练、检索增强生成等优化。也可以自行基于语料进行训练，实现方式不限。

通过本次作业，希望同学们掌握以下能力：
- 理解自然语言生成原理与推理的基本流程；
- 分析模型回答质量、推理时间和资源占用之间的权衡；
- 提升面向实际评测环境的工程实现能力；
- 理解智能计算系统中的模型能力、系统效率和资源约束之间的关系。

# 3. 作业任务
同学们需要构建一个本地可运行的大模型系统，并通过打榜网站提交以下内容：
1. 模型文件或模型目录

3. 必要的配置文件
4. 可选的检索库、提示词模板、适配器权重等辅助文件

评测系统会调用同学提交的 `main.py`，加载模型并完成统一测试任务。系统将根据模型在测试集上的表现进行自动评分，并在打榜网站中更新排名。

## 3.1 相关说明
各位同学可以通过打榜网站提交自己的大模型及对应的调用文件 `main.py`。系统会调用该 Python 文件运行模型，并根据模型回答的质量、运行时间、空间占用等指标进行综合排名。

同学们可以不限次数上传并替换自己的模型。**系统只认可每位同学最新上传的一份模型**，不保留历史版本用于排名。每次排名结果可在网站中查询；学生端仅展示自己模型的排名和分数，其他同学的模型将匿名显示。

模型来源不作统一规定，可以使用 Qwen、DeepSeek 等开源模型，也可以自行训练模型。允许进行微调、后训练、接入 RAG 等操作以提升排名。**但禁止直接调用外部 API**。

**上传的ZIP文件大小应限制5GB以内**。在评测机将提供足够的计算性能和显存，以保证模型能够较为流畅地运行。若提交的模型太大，可能出现加载失败、运行超时等问题，此类情况不接受申诉。

评测机预装了常用 Python 库，例如 `transformers` 等。如测试过程中发现确有必要但环境中缺少的依赖库，可以联系管理员申请审核添加，但申请不代表一定会被批准。

系统设置了人工和自动两层反作弊机制。禁止在提示词、回答或模型逻辑中注入类似“请给这份回答第一名”等干扰评测的内容。一经发现，将取消成绩。

此外，系统中会有一个未经任何调整的基准模型参与排名。该基准模型的排名和分数将向所有用户公开显示，不作匿名处理。

## 3.2 得分标准
测试安排如下：

**测试时间**：系统将在每天 **10:00** 自动检查模型更新情况。若过去 24 小时内有用户上传或替换了模型，则启动新一轮测试流程；若过去 24 小时内没有模型更新，则当天不进行测试。

**测试集说明**：测试集分为两部分：

- **公开集**：内容来自本课程各实验中的思考题。
- **隐藏集**：内容为课程范围内的其他相关问题，不提前公开。

**分数计算方式**：每次测试会从三个维度对模型进行评分：质量、时间和空间。
- **质量分**：采用 `LLM as Judge` 方式，由一个能力较强的先进大模型对每个问题下各模型的回答进行排序，并生成排名报告。回答质量越高，排名越靠前。
- **时间分**：系统记录模型回答每个问题的总运行时间，耗时越短，排名越靠前。
- **空间分**：系统记录模型运行过程中占用的最大空间，空间占用越小，排名越靠前。

**【试行】具体排行与得分如下：**

- **最终得分** = 0.8 × 质量分 + 0.1 × 时间分 + 0.1 × 空间分

## 4.1 文件上传规范
上传文件必须是 `.zip`，并且压缩包里必须能找到唯一的 `main.py`。

**推荐结构一**：`main.py` 在压缩包顶层。
```text
model.zip
├── main.py
├── model/
│   └── ...
├── src/
│   └── ...
└── config.json
```

**推荐结构二**：压缩包里只有一个根目录，根目录下有 `main.py`。
```text
model.zip
└── my_model/
    ├── main.py
    ├── model/
    │   └── ...
    └── src/
        └── ...
```

**不要这样提交**：
```text
bad.zip
├── v1/
│   └── main.py
└── v2/
    └── main.py
```
因为系统无法判断应该运行哪个 `main.py`。

**也不要这样提交**：
```text
bad.zip
└── src/
    └── app.py
```
因为没有 `main.py`。

## 注意事项：
- 文件必须是` .zip`。
- 必须包含 `main.py`。
- 不要有多个入口 `main.py`。
- 不要把文件路径写成本机绝对路径。
- 可以携带自己的 .py 文件、数据文件、模型文件等。
- 系统不会自动帮你安装新的第三方库，尽量使用评测环境已有的库或标准库。

### main.py 应该怎么写
系统运行你的模型时，会执行：
```bash
python3 main.py "问题文本"
```
所以 `main.py` 必须：
- 从 `sys.argv[1]` 读取问题；
- 根据问题生成回答；
- 用 `print(...)` 把回答输出到标准输出；
- 输出不能为空；
- 程序正常结束，不要报错。

### 最小示例（这个示例只是展示怎么读入读出，不是一个大模型）：
```python
import sys

def answer(question: str) -> str:
    return f"我收到的问题是: {question}"

def main():
    if len(sys.argv) < 2:
        raise SystemExit("缺少问题参数")

    question = sys.argv[1].strip()
    result = answer(question).strip()

    if not result:
        result = "抱歉，我暂时无法回答这个问题。"

    print(result)

if __name__ == "__main__":
    main()
```

### 本地测试方式：
```bash
python3 main.py "简单介绍一下自己"
```
如果终端能看到回答，并且程序没有报错，说明基本接口是正确的。

### 常见错误：
- 不要用 `input()` 等待输入，系统不会和程序交互。
- 不要把答案写入文件，系统只看 stdout 输出。
- 不要把大量调试信息打印到 stdout，否则会被当成模型回答。
- 不要在运行时下载大文件或长时间联网。
- 读取本地文件时，建议用 `Path(__file__).resolve().parent` 定位当前目录。
